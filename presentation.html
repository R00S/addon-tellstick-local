Sustainability & Cost

Making Old Hardware Useful Again

The official Home Assistant TellStick add-on was deprecated in December 2024 because the manufacturer stopped maintaining the library. Millions of working TellStick devices faced the e-waste bin.

🌍 Environmental cost of replacing

  • A new Zigbee coordinator: 50–150 g CO₂-eq to manufacture
  • Replacing 10 switches: 500–2 000 g CO₂-eq
  • Working TellStick Duo: already manufactured in 2009–2015
  • Extending its life by 10+ years: near-zero embodied carbon

Why 433 MHz Still Wins

The Forgotten Advantages of 433 MHz

Feature 433 MHz Zigbee / Thread Wi-Fi
Wall-switch batteries None — purely mechanical AA / coin cell, 1–3 yrs Mains or frequent charging
Sensor batteries 3–10 years (AA) 1–2 years (coin cell) Months
Network dependency None — direct RF Needs coordinator/router Needs router + cloud
Device cost €3–15 per outlet €10–30 €10–25
Range (open air) 30–100 m 10–20 m per hop 20–50 m
Protocol updates needed No — protocol is simple Firmware OTA Vendor-dependent
Works during internet outage Yes — always Yes (local) Often no (cloud-dependent)

Key insight: A typical Nexa / KAKU 433 MHz wall socket has no electronics at all on the button side — it's a pure RF transmitter. It will still work in 20 years. Try that with a Zigbee device whose vendor has gone under.

v3.1 Feature

Range Problem → Solved with Mirror / Extender

433 MHz has great range outdoors, but thick concrete walls, basements, and outbuildings can create dead zones. In v3.1 we added a Mirror / Range Extender feature.

How it works

  • Add a second TellStick (Duo, Net, or ZNet) as a "mirror"
  • All ON/OFF/DIM commands are replicated to both units simultaneously
  • Events received by either unit are forwarded to HA
  • Any mix of hardware types works: Duo + Net, ZNet + Duo, etc.
  • No extra configuration — set up via the HA GUI in minutes

📡 Typical deployment

Primary Duo in the living room + a second Net/ZNet unit in the garage. Full 433 MHz coverage across the entire property at zero extra subscription cost.

Roadmap

Future Enhancement Opportunities

🎙 Raw Record & Replay

Capture the exact RF pulse train from any remote — bypass protocol decoding entirely. One button press = one clean signal. Works for unrecognised protocols and eliminates phantom multi-protocol devices.

📡 Full Net/ZNet TX

All 17 telldus-core protocols ported to raw pulse-train encoding for Net/ZNet. Currently arctech and everflourish work; the remaining 12 TX protocols need a pulse encoder per the porting guide.

🔌 Luxorparts Net/ZNet

Luxorparts 50969/50970/50972 work on TellStick Duo via pre-captured codes. Extending this to Net and ZNet requires the raw pulse path above.

📏 Cover Position Tracking

Hasta / Brateck blinds report no position feedback. Timed travel estimation (like most HA cover integrations) would give realistic position without hardware changes.

🏷 MQTT Discovery

Publish device state to an MQTT broker so other home automation platforms (Node-RED, openHAB) can consume 433 MHz events without any HA dependency.

🤖 AI-Assisted Protocol ID

When multiple protocol decoders fire on one button press, use signal timing and bit-pattern analysis to suggest the most likely correct protocol — reducing false-positive device discoveries.

Architecture

How It All Fits Together

The same two-component architecture used by Z-Wave JS and deCONZ — a Docker container for hardware access and a Python integration for HA entities.

1

TellStick USB / LAN

Physical 433 MHz transceiver. Handles RF receive and transmit.

2

telldusd (C daemon in Docker)

Compiled C daemon, USB passthrough via Supervisor. Decodes RF signals.

3

TCP bridge (socat)

Port 50800 (commands) + 50801 (events). Crosses the Docker boundary.

4

HA Custom Integration (Python)

Pure asyncio. Zero native dependencies. Creates entities, config flows, automations.

5

Home Assistant UI & Companion App

Full GUI management — no YAML, no restart, works on Android/iOS.

# 433 MHz button press → HA action in <100 ms Remote button pressed → RF pulse received by TellStick Duo → telldusd decodes → "arctech/selflearning" → TelldusEvents UNIX socket → socat → TCP port 50801 → asyncio reader in client.py → dispatcher.async_dispatch(SIGNAL_RF_EVENT) → switch.py / light.py / sensor.py → HA state machine updated → device trigger fired → automation runs → companion app notified

Why no YAML?

All devices are stored in entry.options["devices"] and re-registered with telldusd at startup. Press a remote → device appears. Click remove → it's gone. No restart needed.

Summary

Key Takeaways

  • AI-first development works — 11 000+ lines of production code, zero manually typed, shipped and used on real hardware.
  • Deprecation ≠ end-of-life — old hardware kept alive, landfill avoided, money saved.
  • 433 MHz is underrated — decade-long battery life, no network dependency, no subscriptions, cheaper devices.
  • Range is not a limitation — the mirror/extender feature turns two cheap sticks into whole-property coverage.
  • Future is open — raw record/replay, full protocol porting, and AI-assisted discovery are all within reach.
